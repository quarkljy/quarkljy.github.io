使用Dockerfile创建docker镜像

Posted on 2019-04-01 In docker Views:
总结Dockerfile创建的docker镜像的常用命令
1. 前言

Dockerfile是一个包含用于组合映像的命令的文本文档。可以使用在命令行中调用任何命令。 Docker通过读取Dockerfile中的指令自动生成映像。docker build命令用于从Dockerfile构建映像。可以在docker build命令中使用-f`标志指向文件系统中任何位置的Dockerfile。

  • 任意路径下的文件Dockerfile，Dockerfile也可以是其他文件名，其他文件名需要指定文件路径。tag没有默认时镜像自动加标签latest

    不指定镜像名字就只有镜像id

  • 当文件名就是Dockerfile时，可以简单使用一下命令构建

Dockerfile文件的命令

主要部分

dockerfile执行build命令时，是从上倒下依次执行的，dockerfile的基本组成部分如下。

主要部分 命令
基础镜像信息 FROM
维护者信息 MAINTAINER
镜像操作指令 RUN、COPY、ADD、EXPOSE、WORKDIR、ONBUILD、USER、VOLUME、ENV等
容器启动时执行指令 CMD、ENTRYPOINT

  • FROM：指定基础镜像，必须为第一个命令

  • MAINTAINER: 维护者信息

  • ENV：设置环境变量

    变量可以传递到shell.sh脚本里

  • RUN：构建镜像时执行的命令

  • ADD：将本地文件添加到容器中，tar类型文件会自动解压(网络压缩资源不会被解压)，可以访问网络资源，类似wget

    可以把shell.sh脚本文件添加进来，最后的用CMD统一执行

  • COPY：功能类似ADD，但是是不会自动解压文件，也不能访问网络资源

  • EXPOSE：指定暴露镜像的端口供主机做映射

  • VOLUME：添加卷，用于指定持久化目录

    有volume卷时，在运行容器时，若没有指定 宿主机文件夹和这些卷对应，那么就会在宿主机的某个文件夹下自动创建一个文件夹与之对应，数据同步。

    例如Dockerfile文件为

      以下情况会使得容器报错

      会让容器内nginx或者mysql程序无法运行

  • USER: 指定运行容器时的用户名或 UID，后续的操作都会使用指定用户。使用USER指定用户时，可以使用用户名、UID或GID，或是两者的组合。当服务不需要管理员权限时，可以通过该命令指定运行用户。并且可以在之前创建所需要的用户

  • ARG：用于指定传递给构建运行时的变量

  • CMD：构建容器后调用，也就是在容器启动时才进行调用。

  • ENTRYPOINT：配置容器，使其可执行化。配合CMD可省去”application”，只使用参数。

    注意！！！！ CMD和ENTRYPOINT的区别

    • CMD的命令会被 docker run 的命令覆盖而ENTRYPOINT不会：如使用CMD [“/bin/bash”]或ENTRYPOINT [“/bin/bash”]后，再使用docker run -ti image启动容器，它会自动进入容器内部的交互终端，如同使用docker run -ti image /bin/bash。
    • 如果启动镜像的命令为docker run -ti image /bin/ps，使用CMD后面的命令就会被覆盖转而执行bin/ps命令，而ENTRYPOINT的则不会，而是会把docker run 后面的命令当做ENTRYPOINT执行命令的参数。
补充

  • 快速进入容器的运行程序的客户端方式

  • docker run 启动容器的命令 末尾最好不要带一起命令，否则可能会被认为是参数传入，导致容器无法启动

  • docker run -v

    若宿主机文件夹1下没有文件，映射到容器里后，容器文件夹2也没有文件，也即是docker把容器文件夹2的指针指向宿主机文件夹1了，不会使用容器内的容器文件夹2，也不会把容器文件夹2下的文件复制到宿主机文件夹1。所以容器文件夹2里的文件不会自动存放到宿主机文件夹1下。若是需要容器文件夹2下的文件，应该先运行容器，运行复制命令
